# 539 量化追蹤台

今彩 539 的個人用「記錄 + 統計 + 驗證」儀表板。每天自動抓取官方開獎，計算各號碼的頻率與遺漏，並以五種方法產生推薦、追蹤命中率，和「純隨機基準」對照。

**線上網站**：<https://sk1010316-cmd.github.io/539666/>

> ⚠️ 先講最重要的一句：539 是公平的獨立隨機抽獎，歷史號碼在數學上對未來**沒有預測力**，長期期望值為負。本專案的價值在於**系統化記錄與驗證**——用數據證明「任何選號法長期都會收斂到隨機」——而不是用來穩定獲利。請當研究與娛樂。

---

## 功能

- **號碼熱力矩陣**：01–39 每格顯示目前遺漏期數與近期出現次數，可切換「遺漏熱度／頻率熱度」上色，滑鼠移上去看歷史最大遺漏。
- **五法推薦引擎**：熱號、冷號回補、均衡、轉移機率、隨機基準，可選近 10／20／30 期視窗。
- **方法回測排行**：用全部歷史做逐期 walk-forward 回測，五法（含隨機）的平均命中並排對照理論值。
- **命中率追蹤**：手動推薦記在本機；系統每天自動替五法各記一組明日推薦，隔日評分並累積在 `data.json`（雲端、跨裝置）。
- **型態統計**：單雙分布、連號分布。

## 五種推薦方法

| 方法 | 定義 | 性質 |
|---|---|---|
| **熱號** | 視窗內出現次數最多的前 3 碼（同分再比長期總次數） | 動能：跟隨近期常開 |
| **冷號回補** | 目前遺漏最久的前 3 碼 | 反向：賭「該開了」（即賭徒謬誤） |
| **均衡** | 長期頻率與遺漏值各自正規化後相加，取分數最高前 3 碼 | 折衷：高頻且久未開 |
| **轉移機率** | 一階馬可夫：統計「本期出現某號後，下期各號出現的條件機率」，把本期 5 號的後續分布加總取前 3 碼 | 條件機率：本期號碼的「歷史後續」 |
| **隨機基準** | 確定性亂數選 3 碼（對照組） | 基準線，用來證明前面方法沒有真正優勢 |

五種都是啟發式，**沒有任何一種具備數學上的真實優勢**。隨機基準是刻意放的對照組。

## 如何解讀回測排行

「方法回測排行」用全部歷史做 walk-forward：對第 *i* 期，只用第 1～*i*−1 期的資料產生推薦，再拿第 *i* 期實際開獎評分，逐期累積。

對照的理論基準（選 3 碼）：

- **每期期望命中碼數** = 3 × 5/39 ≈ **0.385**
- **至少中 1 碼機率** = 1 − C(36,5)/C(39,5) ≈ **34.5%**

正常情況下，五種方法的平均命中都會擠在 0.385 附近，最佳與最差只差雜訊等級的數值。**這個收斂本身就是結論**：在公平抽獎裡，選號策略無法系統性勝過隨機。樣本越大，這個現象越明顯。

## 本機與雲端紀錄的差別

- **我的推薦紀錄（本機）**：你按「產生推薦」時記在瀏覽器 localStorage，換裝置不同步，不會進 repo。
- **雲端每日推薦（累積）**：`build_data.py` 每天自動替五法各記一組明日推薦，隔日評分後寫進 `data.json`，commit 進 repo，**跨裝置可見、長期累積**。這是「無事後諸葛」的真實前瞻紀錄。

---

## 重新部署 / 更新

改完 `index.html` 或 `build_data.py` 後，直接在 GitHub 上覆蓋對應檔案 commit 即可，GitHub Pages 會自動重新部署。`data.json` 的 `live_log` 會被保留累積（`build_data.py` 每次執行會先讀舊檔再追加）。
